# QA Glossary

## A

**Acceptance Criteria**: Conditions that a software product must satisfy to be accepted by a user, customer, or other stakeholder.

**Accessibility Testing (A11y)**: Testing to ensure the application is usable by people with disabilities.

**Ad-hoc Testing**: Informal testing performed without planning or documentation.

**Agile Testing**: Testing practice that follows agile software development principles.

**API (Application Programming Interface)**: Set of protocols and tools for building software applications.

**Assertion**: Statement that a predicate is expected to be true at a particular point in a program.

**Automated Testing**: Use of software tools to execute tests automatically.

## B

**Boundary Value Analysis**: Testing technique that tests boundary values between partitions.

**Bug**: Error, flaw, or fault in software that causes incorrect or unexpected results.

**Bug Tracking**: Process of logging and monitoring bugs in software.

## C

**CI/CD**: Continuous Integration/Continuous Deployment - automated software delivery process.

**Code Coverage**: Measure of how much code is executed during testing.

**Contract Testing**: Verifying interactions between service consumers and providers.

**Cross-Browser Testing**: Testing web applications across different browsers.

**CRUD**: Create, Read, Update, Delete - basic database operations.

**Cypress**: JavaScript end-to-end testing framework.

## D

**Data-Driven Testing**: Testing approach using multiple sets of data.

**Defect**: See Bug.

**Defect Density**: Number of defects per unit of code.

**Deployment**: Process of making software available for use.

## E

**Edge Case**: Problem or situation that occurs at extreme operating parameters.

**End-to-End Testing (E2E)**: Testing complete application flow from start to finish.

**Equivalence Partitioning**: Dividing input data into valid and invalid partitions.

**Exploratory Testing**: Simultaneous learning, test design, and test execution.

## F

**Flaky Test**: Test that sometimes passes and sometimes fails without code changes.

**Functional Testing**: Testing software functions according to requirements.

**Fuzzing**: Providing invalid or random data as input to test robustness.

## G

**GraphQL**: Query language for APIs.

**GUI Testing**: Testing graphical user interface.

## H

**Happy Path**: Default scenario with no exceptional or error conditions.

**Heuristic**: Experience-based technique for problem solving.

## I

**Integration Testing**: Testing combined parts of an application.

**Intermittent Bug**: Bug that occurs sporadically.

## J

**JUnit**: Unit testing framework for Java.

## K

**Keyword-Driven Testing**: Testing using keywords to represent actions.

## L

**Load Testing**: Testing system behavior under expected load.

**Localization Testing (L10n)**: Testing for specific locale or culture.

## M

**Manual Testing**: Testing performed by humans without automation.

**Mock**: Simulated object that mimics real object behavior.

**Mutation Testing**: Modifying program in small ways to test test quality.

## N

**Negative Testing**: Testing with invalid inputs or conditions.

**Non-Functional Testing**: Testing non-functional aspects (performance, security, etc.).

## O

**Open Bug**: Bug that hasn't been fixed yet.

## P

**Page Object Model (POM)**: Design pattern for test automation.

**Penetration Testing**: Simulated cyber attack to find vulnerabilities.

**Performance Testing**: Testing speed, responsiveness, and stability.

**Playwright**: Modern web testing framework.

**Positive Testing**: Testing with valid inputs.

**Pytest**: Testing framework for Python.

## Q

**QA (Quality Assurance)**: Process of ensuring quality in software.

## R

**Regression Testing**: Re-running tests to ensure changes haven't broken existing functionality.

**REST**: Representational State Transfer - architectural style for APIs.

**Risk-Based Testing**: Prioritizing testing based on risk assessment.

**Root Cause Analysis**: Process of finding underlying cause of defects.

## S

**Sanity Testing**: Quick test to verify basic functionality.

**Selenium**: Web browser automation tool.

**Severity**: Degree of impact a bug has on system.

**Smoke Testing**: Preliminary testing to check basic functionality.

**SOAP**: Simple Object Access Protocol - messaging protocol.

**Sprint**: Time-boxed period in Agile development.

**Stub**: Piece of code used to stand in for other code.

**System Testing**: Testing complete integrated system.

## T

**Test Case**: Set of conditions to determine if system works correctly.

**Test Coverage**: Measure of testing completeness.

**Test Data**: Data used for testing purposes.

**Test Plan**: Document describing testing scope, approach, and schedule.

**Test Strategy**: High-level description of test levels and types.

**TestNG**: Testing framework for Java.

**TDD (Test-Driven Development)**: Development approach where tests are written before code.

## U

**UAT (User Acceptance Testing)**: Testing by end users to validate requirements.

**Unit Testing**: Testing individual components in isolation.

**Usability Testing**: Testing ease of use and user experience.

## V

**Validation**: Checking if product meets user needs.

**Verification**: Checking if product meets specifications.

**Visual Regression Testing**: Comparing visual appearance across versions.

## W

**WCAG**: Web Content Accessibility Guidelines.

**White Box Testing**: Testing with knowledge of internal structure.

## X

**XPath**: Language for navigating XML documents.

**XSS (Cross-Site Scripting)**: Security vulnerability in web applications.

## Y

**YAGNI**: "You Aren't Gonna Need It" - principle to avoid unnecessary features.

## Z

**Zero-Day**: Vulnerability unknown to software vendor.
